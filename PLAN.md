# План реализации FindOrigin

Поэтапный порядок действий по реализации Telegram-бота на основании спецификации (PROJECT.md). Стек: Next.js, Vercel; ответ через webhook с быстрым 200 OK.

---

## Этап 1. Инициализация проекта и окружение

1.1. Создать Next.js-проект (App Router), настроить TypeScript, ESLint, структуру папок (app, lib, types).  
1.2. Добавить зависимости: для Telegram Bot API (отправка сообщений), при необходимости — для парсинга (cheerio или аналог), для вызова AI (OpenAI API или выбранный провайдер).  
1.3. Вынести конфигурацию в переменные окружения: `BOT_TOKEN`, `WEBHOOK_SECRET`, URL вебхука, ключи AI. Создать `.env.example` без секретов.  
1.4. Описать типы: структура Telegram update (message, chat.id, text), формат ответа бота (источники + уверенность).

**Результат:** рабочий Next.js-проект, готовый к приёму webhook и вызовам внешних API.

---

## Этап 2. Webhook и быстрый ответ Telegram

2.1. Реализовать API-маршрут (например `app/api/webhook/route.ts`), принимающий только POST.  
2.2. Из запроса извлекать `update.message.chat.id` и `update.message.text` (или данные из инлайн/ссылки — см. ниже). При невалидном теле — сразу возвращать 400.  
2.3. Сразу после валидации отправить клиенту **200 OK** и закрыть соединение; тяжёлую обработку (парсинг, поиск, AI) выполнять асинхронно (фоновая задача/очередь или отдельный вызов).  
2.4. Реализовать отправку ответа пользователю через Telegram Bot API (`sendMessage`): функция в `lib/telegram.ts`, вызов с `BOT_TOKEN`.  
2.5. Настроить вебхук у Telegram (setWebhook) на URL деплоя; при необходимости проверять секрет/подпись от Telegram.  
2.6. Обработать ошибки и таймауты: при падении фоновой задачи отправить пользователю короткое сообщение об ошибке через `sendMessage`.

**Результат:** бот принимает update, сразу отвечает 200, затем асинхронно обрабатывает и пишет результат в чат.

---

## Этап 3. Ввод данных: текст и ссылка на пост

3.1. **Текст.** Если в `message.text` пришёл обычный текст — считать его входом для поиска источника. Санитизация: обрезка длины, базовая защита от инъекций при подстановке в промты/запросы.  
3.2. **Ссылка на Telegram-пост.** Определять ссылки вида t.me/… или telegram.me/… (и при необходимости telegram.dog/…). По одной ссылке на сообщение — достаточно для первой версии.  
3.3. Реализовать извлечение контента поста по ссылке: через Telegram Bot API (если есть доступ к посту) или через парсинг публичной страницы/Telegram Web (с учётом ограничений и ToS). Выделить функцию `getPostContent(link): Promise<string>`.  
3.4. Если контент получить не удалось — отправить пользователю сообщение с просьбой прислать текст или другую ссылку.  
3.5. Итоговый вход для следующих этапов: одна строка текста (либо из message.text, либо извлечённая из поста).

**Результат:** бот умеет принимать и текст, и ссылку на пост; на выходе — один текст для анализа.

---

## Этап 4. Обработка текста: извлечение сущностей

4.1. Нормализовать текст: пробелы, переносы, лишние символы. Сохранить абзацы, если они важны для смысла.  
4.2. Выделить ключевые сущности (без обязательного использования тяжёлого ML на первом этапе):  
- даты (регулярные выражения или простая библиотека под локаль);  
- числа (в т.ч. проценты, суммы);  
- имена собственные (эвристики: заглавные буквы, типичные паттерны);  
- ссылки (URL);  
- короткие ключевые утверждения (например, предложения до 2–3 строк или фразы в кавычках).  
4.3. Реализовать функцию `extractEntities(text): { dates, numbers, names, links, claims }` и покрыть тестами на примерах из спецификации.  
4.4. Сформировать структурированное представление для этапа поиска: список ключевых фраз + сущности. При пустом результате — использовать весь текст как один «запрос».

**Результат:** из пользовательского текста получаем структурированный набор сущностей и утверждений для поиска источников.

---

## Этап 5. Поиск возможных источников

5.1. Определить приоритетные типы источников по спецификации: официальные сайты, новостные сайты, блоги, исследования.  
5.2. Выбрать стратегию поиска:  
- вариант A: поиск через внешний Search API (Google Custom Search, Bing, или другой), фильтрация по типу сайта;  
- вариант B: использование только AI с контекстом (модель сама предлагает типы источников и, при наличии инструментов, «поиск»).  
5.3. Реализовать модуль поиска: на вход — запрос (фразы + сущности), на выход — список кандидатов (URL + заголовок + краткий сниппет). Ограничить число кандидатов (например, 5–10) для следующего этапа.  
5.4. При необходимости — парсинг страниц по URL (cheerio или сервис извлечения текста), получение чистого текста для сравнения с исходным. Учесть таймауты и ошибки доступа.  
5.5. Функция `findCandidateSources(query, entities): Promise<Candidate[]>` с логированием и обработкой ошибок.

**Результат:** по тексту пользователя получаем список кандидатов-источников с URL и контентом (или сниппетами).

---

## Этап 6. AI: сравнение смысла и формирование ответа

6.1. Выбрать модель и API (OpenAI, Claude, локальная и т.д.). Сформулировать системный промт: бот сравнивает **смысл**, а не буквальное совпадение текста.  
6.2. На вход модели: исходный текст пользователя (или ключевые утверждения) + список кандидатов (URL + заголовок + фрагмент текста). Задача: отобрать 1–3 наиболее релевантных источника и дать оценку уверенности (например, в процентах или уровнях).  
6.3. Парсинг ответа модели: извлечь список источников (URL + краткое обоснование) и оценку уверенности. Задать жёсткий формат вывода (JSON или маркированный список в промте).  
6.4. Реализовать функцию `rankSourcesByMeaning(userText, candidates): Promise<RankedSource[]>`.  
6.5. Сформировать итоговое сообщение пользователю: 1–3 ссылки + краткая оценка уверенности. Уложить в лимит Telegram (4096 символов), при необходимости сократить.  
6.6. Отправить сообщение через уже реализованный `sendMessage`.

**Результат:** пользователь получает в чате 1–3 ссылки на возможные источники и оценку уверенности.

---

## Этап 7. Деплой на Vercel и финальная настройка

7.1. Подключить репозиторий к Vercel, настроить сборку Next.js (build command, output).  
7.2. Задать переменные окружения в Vercel: `BOT_TOKEN`, ключи поиска и AI, при необходимости `WEBHOOK_SECRET`.  
7.3. После деплоя обновить webhook Telegram на production URL (https). Проверить, что update приходят и 200 отдаётся быстро.  
7.4. Провести сценарии: только текст; только ссылка на пост; некорректный ввод; ошибки API. Задокументировать известные ограничения (например, типы поддерживаемых ссылок).  
7.5. Добавить минимальное логирование (без секретов): факт прихода update, время ответа, ошибки. При необходимости — мониторинг (Vercel Analytics или внешний).

**Результат:** бот работает в production, стабильно принимает запросы и возвращает ответы в Telegram.

---

## Порядок выполнения

Выполнять этапы последовательно: 1 → 2 → 3 → 4 → 5 → 6 → 7. Этапы 4–6 можно итеративно дорабатывать (качество извлечения сущностей, качество поиска, промты AI) без изменения общего порядка.

---

## Риски и зависимости

- **Ограничения Telegram:** лимиты на длину сообщения, частоту отправки; возможные ограничения на парсинг публичных постов по ссылке.  
- **Внешние API:** квоты и тарифы поиска и AI; необходимость fallback при недоступности сервиса.  
- **Асинхронность:** гарантировать доставку ответа пользователю даже при долгой обработке (очередь, повторные попытки sendMessage).
